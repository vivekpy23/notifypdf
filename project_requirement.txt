================================================================================
PROJECT REQUIREMENT DOCUMENT: GenAI KNOWLEDGE EXTRACTION & NOTIFICATION SYSTEM
================================================================================

1. PROJECT OVERVIEW
-------------------
The GenAI Knowledge Extraction System is an automated pipeline designed to 
transform unread local PDFs into a stream of daily, high-quality insights. 
It leverages Local LLMs and Vector Databases to ensure "passive discovery" 
of knowledge without manual user queries.

2. PROBLEM STATEMENT
--------------------
* Knowledge Silos: Valuable information remains trapped in local PDF storage.
* Information Overload: Manual review of large document sets is time-prohibitive.
* Friction: Traditional RAG systems require a prompt; this system is proactive.

3. SYSTEM ARCHITECTURE
----------------------
[PDF Source] -> [Ingestion] -> [Vector DB] -> [Retrieval] -> [LLM] -> [Mobile]

+-----------------+---------------------------------------+--------------------+
| LAYER           | RESPONSIBILITY                        | TECH STACK         |
+-----------------+---------------------------------------+--------------------+
| Ingestion       | Text extraction & chunking            | LangChain/PyPDF    |
| Storage         | Semantic vector storage               | ChromaDB           |
| Retrieval       | Randomized/Diverse chunk sampling     | Chroma/Ollama      |
| Inference       | Multi-stage extraction & selection    | Ollama (Llama 3+)  |
| Delivery        | Push notifications                    | ntfy.sh            |
+-----------------+---------------------------------------+--------------------+

4. TECHNICAL REQUIREMENTS
-------------------------
* FR1 (Ingestion): Monitor local directories for new PDF content.
* FR2 (Automation): Trigger extraction once every 24 hours autonomously.
* FR3 (Selection): Use a "Judge" prompt to filter the best of N candidates.
* FR4 (Privacy): 100% local processing; no data leaves the local network.
* NFR1 (Quality): Facts must be grounded in source text (No Hallucinations).

5. IMPLEMENTATION STATUS
------------------------
[DONE] PDF Ingestion, Chunking, and Embedding logic.
[DONE] ChromaDB integration and basic ntfy pipeline.
[DONE] Multi-candidate generation and "Selector" logic.
[DONE] Deduplication engine and persistent memory.

6. DESIGN PRINCIPLES
--------------------
1. Quality over Quantity: One great fact is better than ten "dry" ones.
2. Fact-First: No storytelling or fluff; focus on raw insight.
3. Diverse Retrieval: Avoid hitting the same document/topic every day.
4. Minimal Overhead: Low resource usage for 24/7 background operation.

7. FUTURE ROADMAP
-----------------
[DONE] Phase 3: Memory & Deduplication (Avoid repeating facts).
[DONE] Phase 4: Feedback Loop (Thumbs up/down via notification to tune LLM).
Phase 5: Knowledge Graph (Identify entities and connections across PDFs).
Phase 6: Content Gen (Auto-generate threads/summaries from top facts).

8. RISKS & MITIGATIONS
----------------------
* Risk: Extraction of "boring" metadata (e.g., page numbers).
  -> Mitigation: Refine chunking and "Selector" prompt filters.
* Risk: Redundancy.
  -> Mitigation: Semantic similarity check against a "Sent_History" log.
